# sender 验证规则

## 为什么必须验证 sender

自我蒸馏最严重的错误来自 sender 归属反转：把用户说的话当成对方说的，把用户做的事当成对方做的，或把辅导者和被辅导者搞反。根因通常是信任了预处理摘要或语音转录中的 sender 标记，没有回原始数据验证。

## 不可违反规则

1. 凡涉及"谁说了什么""谁做了什么"的判断，必须回原始数据文件验证 sender 字段。
2. verbatim 引用必须标注 `sender=me(已验证)`、`sender=对方(已验证)` 或更具体的已验证 sender。
3. 不能只信预处理脚本、摘要、转写缓存里的 sender 标记。语音转录尤其可能出错。
4. 优先采信用户的自我报告；量化数据只做描述，不做定性终审。
5. 每个主题/批次完成后，抽查 5-10 条关键 verbatim，回原文验证 sender 和上下文。

## 验证流程

1. 记录候选 verbatim 的关键词、时间范围、会话名。
2. 用 `scripts/verify_sender.py` 在合并后的 JSONL 中查找匹配消息。
3. 对照原始字段：`sender`、`is_self`、`talker`、`username`、`conversation`、`timestamp`。
4. 读取前后上下文，确认这句话不是转述、引用或对方复述。
5. 在报告中写 `sender=<name>(已验证)`。

## 引用格式

```markdown
- 证据：[verbatim] 2026-07-07 | source=wechat | sender=me(已验证) | "克制引用"
```

## 高风险场景

- 语音转录消息。
- 大段长消息被摘要后再引用。
- 对话中出现"他说/她说/我说"。
- 一方转述另一方行为。
- 群聊里多人连续发言。
- 预处理脚本生成了精选片段但没有保留原始上下文。
