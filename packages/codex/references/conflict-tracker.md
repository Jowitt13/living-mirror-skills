# CONFLICT 追踪机制

## 原则

矛盾不强行统一。不同来源、不同时间、不同情境中出现不一致的说法时，单独存档，等待更多证据。

## 状态

- `🟢 已解决`：已有足够证据解释矛盾，结论已写回自画像。
- `🟡 待验证`：证据不足，保留多个可能解释。
- `🔴 矛盾加剧`：新证据让矛盾更强，需要优先复核。

## conflicts.md 模板

```markdown
# 矛盾待验证区

> 不同来源、不同时间说法不一致的地方，不强行统一，单独存档。
> 状态：🟢 已解决 / 🟡 待验证 / 🔴 矛盾加剧

---

## CONFLICT-001  <一句话描述>
- 维度：<Part A维度或Part B层级>
- 说法 A：<一个方向>
  - 来源：<source + date>
  - 证据：[verbatim/artifact] <证据>
- 说法 B：<另一个方向>
  - 来源：<source + date>
  - 证据：[verbatim/artifact] <证据>
- 蒸馏器待验证假设：<可以并存的解释，不要强行终审>
- 状态：🟡 待验证
- 首次出现：YYYY-MM
- 本次变化：<新增/更新/关闭原因>
```

## 每次蒸馏开始前

1. 读取 `conflicts.md`。
2. 列出所有未关闭 CONFLICT。
3. 标注本次数据可能验证哪些 CONFLICT。
4. 在批次/主题报告里单独写"CONFLICT 状态"。

## 关闭规则

关闭 CONFLICT 必须满足：

- 有新增证据解释原矛盾。
- 用户复核同意或没有异议。
- 自画像中同步更新为更精确的表述。
- `conflicts.md` 保留关闭记录，不删除历史。
